This guide documents implemented functionality currently present in the repo. It includes the first-run setup flow, the public-api service boundary, and the phase-1 scheduling/calendar system.

1. System Summary

Command is the reusable backoffice and public-service product boundary for a brand-operated installation. It now owns three major runtime layers: the administrator-facing backoffice, the public-api service consumed by brand websites through a server-side BFF, and a shared domain/data model for identity, brands, content, leads, setup, and scheduling.

Command is intentionally split around clear ownership boundaries:

  • Admin Web owns operator workflows, review surfaces, diagnostics, and configuration.
  • Public API owns external/public-site contract behavior.
  • Core packages own shared domain rules that both apps depend on.
  • The Prisma schema is the install-wide source of truth for brand, identity, content, lead, setup, and schedule data.

5. How The Components Interact

5.1 First-Run Setup Flow

  1. A new install without a completed InstallProfile redirects operators to /setup.
  2. The operator unlocks setup using BACKOFFICE_BOOTSTRAP_PASSWORD.
  3. Setup writes the install profile, initial brand, host mapping, brand email metadata, and bootstrap superadmin in one transaction.
  4. The completion screen gives the operator the public integration handoff details needed for website/BFF setup.

5.2 Backoffice Authentication Flow

  1. An operator signs in through /admin/signin.
  2. NextAuth credentials auth delegates to core-auth-backoffice.
  3. If MFA is enabled, the operator must complete the MFA challenge step.
  4. Session state stores role, allowed brand IDs, MFA state, and identity metadata.
  5. Admin pages and admin APIs enforce authorization from that backoffice session context.

5.3 Brand Runtime And Website Integration

  1. Brands are stored in the Brand table.
  2. BrandHost stores allowed public and admin hosts by environment.
  3. Public websites integrate through apps/public-api, not through admin APIs.
  4. The public site BFF sends X-Command-Integration-Key and, when needed, X-Command-Session.
  5. Command resolves brand scope from the integration key rather than trusting raw browser-supplied brand identifiers.

5.4 Content Operations

  1. Prompts, guides, and articles are stored per brand.
  2. Categories are brand-scoped, so the same category names or slugs can exist in different brands.
  3. Admin-web edits this content through internal admin APIs.
  4. Public websites consume published content through the public-api resource endpoints.

5.5 Lead And Service Flow

  1. Contact and chat traffic enter through public-api service endpoints.
  2. Lead and chat activity is captured into the append-only LeadEvent stream.
  3. Optional durable Lead rows provide a summary record when needed.
  4. Dashboard, Leads, and Analytics screens read through core-leads.

5.6 Scheduling Flow

  1. Operators define recurring event series and materialized occurrences.
  2. Named resources represent stages, food spots, market booths, or other schedulable locations.
  3. Schedule participants represent approved/schedulable entities such as entertainers and vendors.
  4. Assignments attach a participant to a resource within an occurrence using either timed-slot or full-day rules.
  5. Conflicts are detected in the backend, reviewed in admin-web, and publish state controls what the public site can see.

8. Current Boundaries And Limitations

  • The public website integration model is explicitly BFF-first. Browser-direct integration is outside the supported pattern.
  • Scheduling phase 1 supports already-approved/schedulable participants only. The future application review/approval module is not implemented yet.
  • The admin calendar uses FullCalendar OSS. Resource-first operations are handled through planner and CRUD workflows rather than premium resource-timeline features.
  • Install env secrets remain operator-owned even though install setup and brand metadata are app-owned.
  • Publish-state and conflict rules are backend-owned. UI surfaces reflect those rules but do not replace them.
