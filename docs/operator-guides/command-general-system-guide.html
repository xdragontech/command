1. System Summary

Command is the reusable backoffice product boundary for brand-operated installations. In this snapshot it already owns the core administrative surface for brand configuration, staff access, client accounts, content operations, lead visibility, analytics, and the public-api contract that brand websites are expected to consume through a server-side BFF layer.

Command is designed around a strong separation of concerns:

  • Admin-web is the backoffice operator UI.
  • Shared core packages own reusable business logic and runtime rules.
  • The database schema is the source of truth for brands, identities, content, and lead data.
  • The public-api contract defines how websites should integrate without directly sharing application runtime code.

5. How The Components Interact

5.1 Backoffice Authentication Flow

  1. An operator signs in through /admin/signin.
  2. NextAuth credentials auth delegates to the backoffice identity logic in core-auth-backoffice.
  3. If MFA is enabled, the operator must complete the MFA challenge step.
  4. Session state stores role, allowed brand IDs, MFA state, and identity metadata.
  5. Admin pages and APIs enforce authorization from that backoffice session context.

5.2 Brand Runtime And Host Resolution

  1. Brands are stored in the Brand table.
  2. Each brand can have multiple hosts in BrandHost for production and preview, across public and admin host kinds.
  3. Runtime host checks use the brand runtime package to decide which install hostnames are valid.
  4. Brand email settings are read live from BrandEmailConfig.

5.3 Content Operations

  1. Prompts and guides are stored per brand.
  2. Categories are also brand-scoped, so the same category name or slug can exist in different brands.
  3. The admin UI edits content through API routes under /api/admin/library/*.
  4. Public websites are expected to read published content through the public-api contract, not through backoffice APIs.

5.4 Lead And Analytics Flow

  1. Lead and chat activity is captured into the append-only LeadEvent stream.
  2. Optional durable lead summary rows live in Lead.
  3. Dashboard, Leads, and Analytics screens read from this event stream using the shared core-leads package.
  4. Counts are deduped by lead/contact identity rules, not simple raw-event totals.

5.5 Website Integration Model

  1. A brand website should use a server-side BFF/proxy.
  2. The BFF sends X-Command-Integration-Key and, for authenticated requests, X-Command-Session.
  3. Command resolves brand scope from the integration principal, not from a raw client-supplied brand key.
  4. The browser should never hold the raw Command session token directly.

8. Current Boundaries And Limitations

  • The public-api is documented as a contract source of truth, but the checked-out repo snapshot is still admin-web heavy.
  • Scheduling/calendar features are not present in this code snapshot and are therefore not described here as implemented functionality.
  • The backoffice intentionally separates runtime diagnostics from editable brand configuration: install envs are read-only in Configs, while brand identity/hosts/email live in Brands.
  • The public website integration model is explicitly BFF-first. Browser-direct integration is outside the supported pattern.
