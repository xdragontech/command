AudienceProduct managers, developers, and technical leads integrating websites with Command
PurposeExplain the public integration contract by feature, including authentication model, sample usage, sample outputs, and recommended integration patterns.
Repository ScopeReflects the OpenAPI contract in packages/contracts-openapi/command-public-api.v1.yaml at commit b13028c.
This guide is based on the public-api contract source of truth. In this repo snapshot the contract is authoritative even though the dedicated public-api app source is not fully represented here.

1. Integration Model

Websites should integrate with Command through a server-side BFF or proxy. The browser should not call Command directly.

  1. The website server stores the Command session token.
  2. The website server forwards requests to Command with the integration key and, when needed, the Command session token.
  3. The browser only talks to the website’s own BFF routes.
Required headers
X-Command-Integration-Key: server-to-server credential proving the website is allowed to call the install.
X-Command-Session: opaque end-user session token issued by Command and forwarded by the BFF.

3. Authentication Feature Details

3.1 Register

What it does: Creates a brand-scoped external account and starts the verification flow.

POST /v1/auth/register
{
  "email": "user@example.com",
  "password": "StrongPassword123",
  "name": "Grant Rogers"
}
{
  "ok": true,
  "verificationRequired": true
}

3.2 Login

What it does: Authenticates a user and returns both account data and an opaque session token for BFF storage.

POST /v1/auth/login
{
  "email": "user@example.com",
  "password": "StrongPassword123"
}
{
  "ok": true,
  "session": {
    "token": "cmd_sess_opaque_token",
    "expiresAt": "2026-03-30T21:15:00.000Z"
  },
  "account": {
    "id": "extusr_123",
    "brandKey": "xdragon",
    "email": "user@example.com",
    "name": "Grant Rogers",
    "status": "ACTIVE",
    "emailVerified": true,
    "createdAt": "2026-03-20T19:10:00.000Z",
    "updatedAt": "2026-03-23T17:40:00.000Z",
    "lastLoginAt": "2026-03-23T21:15:00.000Z"
  }
}

3.3 Session Inspection

What it does: Lets the BFF ask Command whether the current stored session is still valid.

GET /v1/auth/session
Headers:
  X-Command-Integration-Key: <integration-key>
  X-Command-Session: <opaque-session-token>
{
  "ok": true,
  "account": {
    "id": "extusr_123",
    "brandKey": "xdragon",
    "email": "user@example.com",
    "name": "Grant Rogers",
    "status": "ACTIVE",
    "emailVerified": true,
    "createdAt": "2026-03-20T19:10:00.000Z",
    "updatedAt": "2026-03-23T17:40:00.000Z",
    "lastLoginAt": "2026-03-23T21:15:00.000Z"
  }
}

3.4 Logout

What it does: Invalidates the current session so the BFF can clear its local session envelope.

3.5 Verify Email

What it does: Completes email verification using the token from the brand email flow.

POST /v1/auth/verify-email
{ "token": "raw-email-verification-token" }
{ "ok": true, "verified": true }

3.6 Forgot Password And Reset Password

What they do: Start and finish the password reset flow.

POST /v1/auth/password/forgot
{ "email": "user@example.com" }
{ "ok": true }
POST /v1/auth/password/reset
{
  "token": "raw-reset-token",
  "password": "NewStrongPassword123"
}
{ "ok": true }

4. Account Feature Details

4.1 Fetch Current Account

Use GET /v1/account/me with both required headers to retrieve the current external user profile.

4.2 Update Current Account

Current v1 scope is intentionally narrow: only display name updates are in contract.

PATCH /v1/account/me
Headers:
  X-Command-Integration-Key: <integration-key>
  X-Command-Session: <opaque-session-token>
Body:
{
  "name": "Updated Display Name"
}
{
  "ok": true,
  "account": {
    "id": "extusr_123",
    "brandKey": "xdragon",
    "email": "user@example.com",
    "name": "Updated Display Name",
    "status": "ACTIVE",
    "emailVerified": true,
    "createdAt": "2026-03-20T19:10:00.000Z",
    "updatedAt": "2026-03-23T22:05:00.000Z",
    "lastLoginAt": "2026-03-23T21:15:00.000Z"
  }
}

6. Multi-Step Integration Examples

6.1 Registration To First Authenticated Resource Read

  1. Call POST /v1/auth/register.
  2. Wait for the user to verify email through the tokenized email flow.
  3. Call POST /v1/auth/login and store session.token in the BFF session layer.
  4. Call GET /v1/auth/session to confirm the session.
  5. Call GET /v1/resources/guides or /v1/resources/prompts using the same session token.

6.2 Forgot Password Recovery

  1. Call POST /v1/auth/password/forgot with the user email.
  2. Deliver the reset link/token through the brand email flow.
  3. Call POST /v1/auth/password/reset with the raw reset token and new password.
  4. Call POST /v1/auth/login with the new password to start a fresh session.

6.3 Logged-In Profile Update

  1. Login and store the returned opaque session token.
  2. Call PATCH /v1/account/me with the new display name.
  3. Immediately refresh local website session state from the updated account response or a fresh GET /v1/auth/session.

7. Error Handling

All documented error responses use the same basic shape:

{
  "ok": false,
  "error": "Human-readable error message"
}

Common status codes:

  • 400: malformed request or bad token
  • 401: missing/invalid integration key or session token
  • 404: resource not found
  • 409: conflicting create/state request
  • 422: validation failure or unsupported state transition

8. Integration Recommendations

  • Keep the integration key only on the server side.
  • Keep the Command session token only in the website’s BFF session layer.
  • Do not let the browser call Command directly.
  • Treat the OpenAPI file as the contract source of truth when implementing or validating endpoints.
  • Assume the website is brand-scoped through the integration principal, not through arbitrary browser-provided brand fields.
