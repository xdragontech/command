X Dragon Command Back Office

API Integration Guide
AudienceProduct managers, developers, and technical leads integrating websites with Command
PurposeExplain the public integration contract by feature, including authentication model, sample usage, sample outputs, and recommended integration patterns.
Repository ScopeReflects the implemented public-api app and OpenAPI contract in the current checked-out source as of 2026-03-23.
Source of truth for the wire contract is packages/contracts-openapi/command-public-api.v1.yaml. This guide explains the contract in operator/developer terms and uses the actual deployed Next.js path shape: /api/*.

1. Integration Model

Websites should integrate with Command through a server-side BFF or proxy. The browser should not call Command directly.

  1. The website server stores the Command session token.
  2. The website server forwards requests to Command with the integration key and, when needed, the Command session token.
  3. The browser only talks to the website’s own BFF routes.
Required headers
X-Command-Integration-Key: server-to-server credential proving the website is allowed to call this install.
X-Command-Session: opaque end-user session token issued by Command and forwarded by the BFF for authenticated flows.

3. Operational Endpoints

3.1 Health

What it does: confirms the service is reachable.

GET /api/healthz
{
  "ok": true,
  "service": "command-public-api"
}

3.2 Readiness

What it does: confirms required envs, database connectivity, brand resolution, and brand email readiness.

GET /api/readyz
{
  "ok": true,
  "checks": {
    "database": "ok",
    "integrations": "ok",
    "brands": "ok"
  }
}

4. Authentication Feature Details

4.1 Register

What it does: Creates a brand-scoped external account and starts the verification flow.

POST /api/v1/auth/register
Headers:
  X-Command-Integration-Key: <integration-key>
Body:
{
  "email": "user@example.com",
  "password": "StrongPassword123",
  "name": "Grant Rogers"
}
{
  "ok": true,
  "verificationRequired": true
}

4.2 Login

What it does: Authenticates a user and returns both account data and an opaque session token for BFF storage.

POST /api/v1/auth/login
Headers:
  X-Command-Integration-Key: <integration-key>
Body:
{
  "email": "user@example.com",
  "password": "StrongPassword123"
}
{
  "ok": true,
  "session": {
    "token": "cmd_sess_opaque_token",
    "expiresAt": "2026-03-30T21:15:00.000Z"
  },
  "account": {
    "id": "extusr_123",
    "brandKey": "xdragon",
    "email": "user@example.com",
    "name": "Grant Rogers",
    "status": "ACTIVE",
    "emailVerified": true,
    "createdAt": "2026-03-20T19:10:00.000Z",
    "updatedAt": "2026-03-23T17:40:00.000Z",
    "lastLoginAt": "2026-03-23T21:15:00.000Z"
  }
}

4.3 Session Inspection

What it does: Lets the BFF ask Command whether the current stored session is still valid.

GET /api/v1/auth/session
Headers:
  X-Command-Integration-Key: <integration-key>
  X-Command-Session: <opaque-session-token>
{
  "ok": true,
  "account": {
    "id": "extusr_123",
    "brandKey": "xdragon",
    "email": "user@example.com",
    "name": "Grant Rogers",
    "status": "ACTIVE",
    "emailVerified": true,
    "createdAt": "2026-03-20T19:10:00.000Z",
    "updatedAt": "2026-03-23T17:40:00.000Z",
    "lastLoginAt": "2026-03-23T21:15:00.000Z"
  }
}

4.4 Logout

What it does: Invalidates the current session so the BFF can clear its own local session envelope.

POST /api/v1/auth/logout
Headers:
  X-Command-Integration-Key: <integration-key>
  X-Command-Session: <opaque-session-token>

4.5 Verify Email

What it does: Completes email verification using the token sent by the brand email flow.

POST /api/v1/auth/verify-email
Headers:
  X-Command-Integration-Key: <integration-key>
Body:
{ "token": "raw-email-verification-token" }
{ "ok": true, "verified": true }

4.6 Forgot Password And Reset Password

What they do: Start and finish the password reset flow.

POST /api/v1/auth/password/forgot
Headers:
  X-Command-Integration-Key: <integration-key>
Body:
{ "email": "user@example.com" }
{ "ok": true }
POST /api/v1/auth/password/reset
Headers:
  X-Command-Integration-Key: <integration-key>
Body:
{
  "token": "raw-reset-token",
  "password": "NewStrongPassword123"
}
{ "ok": true }

5. Account Feature Details

5.1 Fetch Current Account

Use GET /api/v1/account/me with both required headers to retrieve the current external user profile.

5.2 Update Current Account

Current v1 scope is intentionally narrow: only display-name updates are in contract.

PATCH /api/v1/account/me
Headers:
  X-Command-Integration-Key: <integration-key>
  X-Command-Session: <opaque-session-token>
Body:
{
  "name": "Updated Display Name"
}
{
  "ok": true,
  "account": {
    "id": "extusr_123",
    "brandKey": "xdragon",
    "email": "user@example.com",
    "name": "Updated Display Name",
    "status": "ACTIVE",
    "emailVerified": true
  }
}

8. Service Flow Details

8.1 Contact

Purpose: Accept a public contact request, write lead records, and attempt notification delivery.

POST /api/v1/contact
Headers:
  X-Command-Integration-Key: <integration-key>
Body:
{
  "name": "Casey Smith",
  "email": "casey@example.com",
  "phone": "555-000-1111",
  "message": "I want more information about vendor opportunities."
}
{
  "ok": true,
  "id": "lead_evt_123",
  "notification": "sent"
}

8.2 Chat

Purpose: Process a public website chat turn and optionally capture follow-up lead information.

POST /api/v1/chat
Headers:
  X-Command-Integration-Key: <integration-key>
Body:
{
  "messages": [
    { "role": "user", "content": "I want to book a booth for opening weekend." }
  ],
  "lead": {
    "name": "Casey Smith",
    "email": "casey@example.com"
  }
}
{
  "ok": true,
  "reply": "We can help with that. Which date and booth type are you interested in?",
  "lead": {
    "captured": true
  }
}

9. Multi-Step Integration Examples

9.1 Registration To First Authenticated Resource Read

  1. Call POST /api/v1/auth/register.
  2. Wait for the user to verify email through the tokenized email flow.
  3. Call POST /api/v1/auth/login and store session.token in the BFF session layer.
  4. Call GET /api/v1/auth/session to confirm the session.
  5. Call GET /api/v1/resources/guides or /api/v1/resources/prompts using the same session token.

9.2 Public Schedule Rendering

  1. Call GET /api/v1/schedule/calendar with a bounded date range and the integration key.
  2. Render the returned entries in the website calendar UI.
  3. Use GET /api/v1/schedule/list for the companion list/cards/filters view.
  4. Only published assignments are returned, so draft changes remain private until operators publish them.

9.3 Forgot Password Recovery

  1. Call POST /api/v1/auth/password/forgot with the user email.
  2. Deliver the reset link/token through the brand email flow.
  3. Call POST /api/v1/auth/password/reset with the raw reset token and new password.
  4. Call POST /api/v1/auth/login with the new password to start a fresh session.

10. Implementation Rules

  • Do not call Command directly from the browser.
  • Store the raw Command session token only in the BFF/server layer.
  • Use one integration key per public site and keep it brand-scoped in v1.
  • Treat /api/readyz as an operator/runtime check, not a public user feature.
  • Use published schedule feeds for public calendars. Draft schedule data is intentionally not exposed.
