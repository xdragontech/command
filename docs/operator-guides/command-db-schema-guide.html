This schema now includes install setup state and the phase-1 scheduling domain alongside the earlier identity, brand, content, and lead tables.

1. Schema Overview

The schema is organized around eight main domains:

  1. Legacy user/auth tables
  2. Brand and runtime configuration
  3. Scheduling
  4. Install setup state
  5. Backoffice operator identity
  6. External/client identity
  7. Content (prompts, guides, and articles)
  8. Leads and lead events
The most important structural rule is brand scoping. Most operational tables are anchored under Brand, which is the install’s primary tenant/context boundary.

3. Data Hierarchy

  • InstallProfile records whether the install is initialized and which brand is primary.
  • Brand sits at the center of runtime configuration, scheduling, content, public identities, and lead data.
  • Backoffice users gain brand access through the join table BackofficeUserBrandAccess.
  • External users are directly brand-scoped.
  • Scheduling hangs under brand through series, occurrences, resources, participants, and assignments.
  • Prompts, articles, and leads are also brand-scoped or optionally brand-linked.

4. Legacy Auth Domain

The old shared User/LoginEvent auth domain was retired and removed from the active schema on 2026-03-30. Command now uses the separate Backoffice* and External* identity domains only.

12. Relationship Notes By Feature

  • InstallProfile → Brand: the install can point to one primary brand but brands remain independent runtime records.
  • Brand → Host mapping: each brand can own multiple hosts, separated by environment and host kind.
  • Brand → Scheduling: series, occurrences, resources, participants, and assignments all remain brand-scoped.
  • ScheduleEventSeries → ScheduleEventOccurrence: one-to-many, with explicit materialized occurrences for recurrence safety.
  • ScheduleEventOccurrence / ScheduleResource / ScheduleParticipant → ScheduleAssignment: assignments are the concrete join point for scheduled behavior.
  • BackofficeUser ↔ Brand: staff access is many-to-many through BackofficeUserBrandAccess.
  • Brand → External users: public/client accounts are directly brand-scoped.
  • Prompt / Article content: both can optionally be unscoped at the schema level, but the backoffice is built around brand-scoped management.
  • LeadEvent → Lead: lead events may point to a durable lead row when one exists, but analytics still read the event stream.

14. Review Notes

  • The legacy shared auth domain has been removed. Backoffice and external identities are the only active auth domains in the schema.
  • The public-api contract depends on the external identity, brand, content, lead, and scheduling parts of this schema.
  • The scheduling domain is intentionally phase-1 shaped: it supports approved/schedulable entities now, while future application-review linkage is deferred.
  • InstallProfile is the explicit install-initialization marker. Setup should not be inferred indirectly from “no brands” or “no users”.
