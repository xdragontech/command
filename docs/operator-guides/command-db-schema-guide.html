This schema guide reflects the current checked-out schema only. Scheduling/calendar tables discussed in later planning are not present in this snapshot.

1. Schema Overview

The schema is organized around six main domains:

  1. Legacy user/auth tables
  2. Brand and runtime configuration
  3. Backoffice operator identity
  4. External/client identity
  5. Content (prompts and guides)
  6. Leads and lead events
The most important structural rule in this schema is brand scoping. Many operational tables are anchored under Brand, which is the installation’s primary tenant/context boundary.

3. Data Hierarchy

  • Install-wide identity: legacy User tables and backoffice operator tables
  • Brand sits at the center of runtime configuration and brand-scoped operational data
  • Backoffice users gain brand access through the join table BackofficeUserBrandAccess
  • External users are directly brand-scoped
  • Prompts, guides, guide categories, leads, and lead events can all be brand-scoped

Important content rule: categories and guide categories are brand-scoped, which means the same category name can be reused in different brands without collision.

9. Domain: Leads

TablePurposeKey Relationships / Notes
LeadDurable lead summary rowOptional brand scope; stores source, name, email, IP, and raw payload.
LeadEventAppend-only lead event streamSource of truth for lead analytics and lead history; may optionally link back to Lead.

The code explicitly treats LeadEvent as the analytics source of truth. Dashboard and analytics counts are deduped from event data rather than simple raw-event totals.

10. Relationship Notes By Feature

  • Brand → Host mapping: each brand can own multiple hosts, separated by environment and host kind.
  • Brand → External users: public/client accounts are directly brand-scoped.
  • BackofficeUser ↔ Brand: staff access is many-to-many through BackofficeUserBrandAccess.
  • Prompt / Article content: both can optionally be unscoped, but the backoffice UI is built around brand-scoped management.
  • LeadEvent → Lead: lead events may point to a durable lead row when one exists, but analytics still read the event stream.

12. Review Notes

  • The schema still contains legacy user/auth tables alongside newer backoffice and external identity domains.
  • The public-api contract assumes external identity and brand-scoped resources from this schema.
  • No scheduling/calendar schema exists in this branch, so seeded scheduling smoke work cannot be validated against this snapshot alone.
